FLUXO COMPLETO DE UMA BATALHA (estado atual)

  ContentView / BattleDebugLauncher
        |
        v
  BattleView
        |  init(player, opponent?) -> cria BattleViewModel
        v
  BattleViewModel.init
        | engine = BattleEngine()
        | evaluator = TimingEvaluator(balance: engine.balance)
        | opponentChooser = BattleAIDecisionProvider()   (FIX: provider cria SystemRandomNumberProvider)
        | state = engine.makeInitialState(player, opponent)
        v
  Phase.choosingAction
        |
        |  user toca "Attack" em BattleActionControls
        v
  BattleViewModel.chooseAction(.attack)
        | valida validPlayerActions (do engine)
        | phase = .timing(.attack)
        v
  TimingBar (SwiftUI)
        |  TimelineView anima position em [0..1] (ida e volta a cada 2s)
        |  user toca no bar ou em "Confirm timing"
        v
  BattleViewModel.confirmTiming(normalizedPosition)
        | phase = .resolving
        | timing = TimingEvaluator.evaluate(position, agility)
        | playerDecision = BattleDecision(action, timing)
        | opponentDecision = opponentChooser.chooseDecision(...)
        |   (BattleAIDecisionProvider.delegate -> SimpleBattleAI.chooseDecision)
        | resolution = try engine.resolveRound(state, pDecision, oDecision)
        | state, roundResult atualizados
        | phase = .finished(outcome)  ou  .showingRoundResult
        v
  BattleView.body switch sobre Phase
        | choosingAction   -> "Choose your action..."
        | timing(action)   -> VStack{Title, Hint, TimingBar}
        | resolving        -> ProgressView("Resolving round…")
        | showingRoundResult -> RoundSummaryView
        | finished         -> BattleEndOverlay (overlay na view)
        v
  user toca "Continue" (showingRoundResult) ou "Play again" (finished)
        | continueBattle() -> phase = .choosingAction
        | restart()        -> state reinicializado


2. PONTOS FORTES (DEVEM SER PRESERVADOS)
------------------------------------------------------------------------------

P1. Domínio puro, isolado de SwiftUI.
    Verificado: nenhum arquivo em Domain/Battle/*.swift importa SwiftUI, UIKit,
    AVFoundation, Foundation Models, etc. Esta é a maior força da arquitetura
    atual e o motivo pelo qual o engine pode ser testado sem UI.

P2. BattleEngine.resolveRound é uma função "throws-pura".
    Mesmo com `var combatant = state.player` para mutar, a função é determinística
    dado (state, playerDecision, opponentDecision). Não chama random, não chama
    UI, não tem side effects além de retornar BattleRoundResolution.
    Isso é compatível com replay, telemetria e snapshots.

P3. RoundResult é um snapshot completo.
    Já carrega: decisão completa de ambos combatentes, hp before/after, energia
    before/after, energia gasta/recuperada, dano aplicado, outcome.
    Não vai precisar ser reescrito para suportar rítmo — só talvez estendido
    com novos campos (tempo de input, score, lista de acertos).

P4. TimingEvaluator já é domínio, não UI.
    Avalia normalizedPosition + agility -> TimingResult. Sem Date(), sem
    SwiftUI. Para rítmo, este é o ponto onde precision vira score; a abstração
    está pronta.

P5. SimpleBattleAI é parametrizado por RandomNumberProviding.
    Os testes SimpleBattleAITests já injetam SequenceRandom e checam determinismo.
    O ponto de injeção existe. Falta apenas a ViewModel propagá-lo (ver P-1).

P6. BattleBalance concentra tuning.
    Todas as fórmulas de HP, energia, dano, redução, timing e pesos de CPU estão
    em um único struct Sendable. Trocar balance é criar outro static let.

P7. Engines de batalha separados por papel.
    BattleEngine (regras), TimingEvaluator (precisão), SimpleBattleAI (CPU),
    DeterministicStatCalculator (geração de stats), BattleBalance (tuning).
    Cada um testado independentemente. Boa base para crescer.

P8. Swift 6 / Sendable já preparado onde importa.
    BattleState, CombatantState, BattleDecision, RoundResult, BattleBalance,
    TimingResult, BattleAction: todos Sendable. BattleAIDecisionProvider é
    class (não Sendable), mas BattleViewModel é @MainActor.

P9. Decisões registradas em TESTING/COMBAT/TECH_DECISIONS.
    Mesmo desatualizadas (ver B-6), já existe o hábito de documentar decisões
    técnicas. Continuar o hábito será mais barato que improvisar.

CRÍTICO
----------------------------------------------------------------

C-1. Fonte de aleatoriedade não é capturada pela BattleViewModel
    Arquivo: snap-battle/Features/Battle/BattleViewModel.swift:43 e :123-128
    Comportamento atual:
        opponentChooser = opponentChooser ?? BattleAIDecisionProvider()
        final class BattleAIDecisionProvider {
            private var ai = SimpleBattleAI(random: SystemRandomNumberProvider())
        }
    Hoje o RandomNumberProviding é instanciado dentro de BattleAIDecisionProvider
    e nunca é exposto. Logo, a batalha rodada na tela NÃO é reproduzível: o
    engine é determinístico-dado-inputs, mas a sequência de decisões da CPU é
    gerada por um SystemRandomNumberGenerator interno que ninguém controla.
    Risco: sem seed injetado, replay e debug de "por que a CPU fez aquilo?"
    são impossíveis; testes de ViewModel não conseguem cobrir a AI real
    (eles usam CountingChooser como stub).
    Recomendação:
        a) Tornar a fonte de aleatoriedade parte do init do BattleViewModel.
        b) BattleAIDecisionProvider deve aceitar um RandomNumberProviding
           (injetado) em vez de instanciar SystemRandomNumberProvider.
        c) Para o app final, considerar um BattleSeed no BattleState que
           também governa CPU e (no futuro) geração de ritmo.

C-2. Documentação crítica desatualizada em relação ao código
    Arquivos: docs/COMBAT.md:1-19 e docs/TESTING.md:115
    Comportamento atual:
        COMBAT.md: "Status: Not implemented." — diz que combate não existe.
        TESTING.md linha 115: "No combat tests because combat is not
        implemented." — completamente errado, há 5 arquivos de teste de combate.
        ROADMAP.md: Phase 4 Combat Prototype marcado "Exploration" — também
        já há batalha funcional.
        TECH_DECISIONS.md "Current Architecture Notes" não menciona Battle/,
        BattleEngine, BattleViewModel, etc.
    Risco: contribuidor futuro ou o próprio usuário perde 30 minutos lendo
    docs antes de descobrir o motor já existente. Decisões técnicas tomadas
    em código ficam invisíveis para revisão e para IA assistiva.
    Recomendação:
        a) Atualizar COMBAT.md para refletir o sistema atual; listar as
           decisões já embutidas (action+timing, resolução simultânea,
           energia como recurso, balance tuning).
        b) Atualizar ROADMAP.md: Phase 4 já atingiu "Implemented at PoC
           level"; abrir Phase 6 "Rhythm-based combat" (ver item 5).
        c) Atualizar TESTING.md com a nova árvore de testes de combate.
        d) Atualizar TECH_DECISIONS.md com decisões recentes (BattleEngine
           throws-puro, TimingEvaluator como domínio, batalha síncrona por
           enquanto).

ALTO
----------------------------------------------------------------

A-1. Máquina de estados vive no ViewModel, não no domínio
    Arquivo: snap-battle/Features/Battle/BattleViewModel.swift:10-16
    Comportamento atual:
        enum Phase { choosingAction | timing(action) | resolving
                     | showingRoundResult | finished(outcome) }
        Quem garante "só chamar chooseAction quando phase=.choosingAction?"
        é o ViewModel, via guard de método. O engine não conhece Phase.
    Hoje funciona porque há 1 source (o ViewModel) e 1 consumidor (a View).
    Risco: ao introduzir ritmo, a fase precisa ser representável no domínio
    para que se possa pedir input com janelas ("tempo 0..0.5, abrir janela
    em 0.5") e para que dois observadores (View + áudio) enxerguem a mesma
    fase sem race.
    Recomendação:
        Mover Phase do domínio para snap-battle/Domain/Battle/BattlePhase.swift
        (puro, Equatable + Sendable). BattleViewModel armazena
        phase: BattlePhase e a View continua observando.

A-2. Validação de ações válidas duplicada
    Arquivos:
        Services/Battle/BattleEngine.swift:23-28
        Services/Battle/SimpleBattleAI.swift:22-27
    Comportamento atual:
        O método validActions(for:) existe em BattleEngine e é DUPLICADO,
        linha por linha, em SimpleBattleAI como método privado. Nada
        impede que a regra divergir (ex.: alguém adiciona "special"
        condicional no engine e esquece a AI).
    Risco: inconsistência entre UI e AI (a UI pode bloquear um botão que
    a CPU executa, ou vice-versa). Hoje ambas as cópias são idênticas, mas
    é um bug latente.
    Recomendação:
        Extrair para uma função pura (ex.: Balance.validActions(for:)) ou
        tornar a AI dependente do engine:
            BattleActionChoosing.chooseDecision recebe um GameRules-like
            access a validActions.
        A solução mais limpa é mover a checagem para dentro do engine e
        expor como static / Balance-bound.

A-3. Timing = detalhe de UI hoje; vai precisar virar detalhe do engine
    Arquivos:
        Features/Battle/Components/TimingBar.swift:53-58
        Features/Battle/BattleViewModel.swift:64-88
    Comportamento atual:
        A View decide QUANDO pedir input (TimelineView anima e o usuário
        toca), então passa position normalizada para confirmTiming, que
        então chama TimingEvaluator.
        Em outras palavras: o "momento da captura" é decidido pela SwiftUI.
        O domínio não emite nenhuma janela.
    Risco: para combate rítmico, a janela precisa ser ditada pelo domínio
    (o ritmo da música / o pulso do combatente), e a UI precisa seguir.
    Se o domínio emitir "input window aberta em t=0.5s, fecha em t=0.7s,
    precisa-se acertar nota X", então TimingEvaluator passa a ser
    "avalie uma sequência de TimingInput dentro de RhythmPattern" — uma
    evolução natural, mas só após inverter o sentido da conversa
    (engine -> UI em vez de UI -> engine).
    Recomendação:
        a) Introduzir uma FSM de domínio (A-1).
        b) Quando entrar em .awaitingRhythmInput(pattern), o engine já
           publica a janela; a View só observa e captura inputs.
        c) TimingEvaluator hoje é uma função que poderia ser reusada
           por input temporal; verificar compatibilidade.

A-4. Modificadores de combate fundidos com base
    Arquivo: snap-battle/Services/Battle/BattleEngine.swift:120-131
              snap-battle/Domain/Battle/BattleBalance.swift:71-100
    Comportamento atual:
        damage(...) lê baseAttackDamage, soma power/divisor, multiplica por
        timing percent. Se quisermos "queimadura dobra dano" ou "+shield
        30%", não há como; tudo está dentro de BattleBalance como fórmula
        fechada.
    Risco: ao adicionar status (burn, shield, stun...), a fórmula
    grow virará spaghetti em damage(...). Ou criamos uma acumulação de
    multiplicadores (boa escolha) ou uma pilha de CombatEffect (idem).
    Recomendação:
        a) Não aplicar agora (status ainda não existem).
        b) Quando entrar: introduzir CombatEffect e CombatantModifiers
           capturados antes de damage(...).
        c) Manter BattleBalance como tabela BASE; modificadores em runtime
           ficam fora.

A-5. BattleAction enum fechado impede evolução natural
    Arquivos:
        Domain/Battle/BattleAction.swift
        Services/Battle/BattleEngine.swift:112-118
        Features/Battle/Components/BattleActionControls.swift:35-45
    Comportamento atual:
        enum BattleAction: String, CaseIterable { attack, defend, charge }
        Toda a UI, AI e balance iteram switch sobre os 3 cases.
    Risco: adicionar ".special" ou ".counterattack" vai exigir mudar
    BattleAction (cobre Domain/Services/UI/Testes). É esperado para um
    PoC, mas com ritmo precisamos de ações parametrizadas (custo,
    carga, tempo).
    Recomendação:
        a) Para o ritmo, NÃO adicionar mais cases no BattleAction atual.
        b) Considerar separar "o que o jogador intenta" (BattleCommand
           ou ActionPattern) de "o tipo genérico da intenção" (BattleAction).
           O motor avalia um Command -> gera decision -> resolve.
        c) Esta é uma mudança para o passo 5 do plano (seção 6).

MÉDIO
----------------------------------------------------------------

M-1. Phase.resolving é um placeholder síncrono
    Arquivo: snap-battle/Features/Battle/BattleViewModel.swift:66-87
    Comportamento atual:
        confirmTiming() faz phase = .resolving e IMEDIATAMENTE resolve
        (engine síncrono) e segue para showingRoundResult/finished.
        O .resolving nunca é visto pelo usuário como estado distinto em
        prática — só aparece como "ProgressView" se a render cair entre
        os dois assigns, o que em prática pode nem acontecer.
    Risco: para rítmo, .resolving precisará ser uma fase observável,
    longa o suficiente para sincronizar com animação+áudio, e que possa
    ser pulada com "skip animations" para acessibilidade.
    Recomendação:
        a) Quando o motor virar assíncrono (ou pelo menos baseado em
           eventos), .resolving vira o intervalo entre
           "decisão tomada" e "resultado produzido".
        b) Hoje pode-se manter síncrono — é apenas um marcador.

M-2. selectedAction duplica a informação que já existe em Phase
    Arquivo: snap-battle/Features/Battle/BattleViewModel.swift:27, 58-79
    Comportamento atual:
        Phase.timing(BattleAction) já carrega a ação; também há o
        campo selectedAction: BattleAction?. Após o resolve, é zerado.
        A View, no entanto, lê apenas roundResult?.player.decision.action
        (não usa selectedAction) — então selectedAction é puramente
        interno e redundante com o phase.
    Risco: dois caminhos para a mesma informação. Sutil, mas propenso
    a divergir (ex.: alguém esquece de atualizar um dos dois).
    Recomendação:
        a) Derivar selectedAction via computed property:
           var selectedAction: BattleAction? {
               if case .timing(let action) = phase { return action }
               return nil
           }
        b) Remover o campo @Observable.

M-3. Battle.AIDecisionProvider não é testável através do ViewModel
    Arquivo: snap-battle/Features/Battle/BattleViewModel.swift:123-129
    Comportamento atual:
        O provider embute SystemRandomNumberProvider, então mesmo
        injetando um stub no init (CountingChooser), o caminho real
        do SimpleBattleAI não é exercitado pelos testes de ViewModel.
    Risco: mudanças nos pesos da AI passam batido pelo BattleViewModelTests.
    Recomendação:
        Inverter: o provider recebe um RandomNumberProviding por init.

M-4. Acesso direto a roundResult?.decision.* na View
    Arquivo: snap-battle/Features/Battle/BattleView.swift:16-27
    Comportamento atual:
        A View extrai decisão/energia/HP de roundResult?.player/opponent
        para alimentar HUDs. Hoje funciona porque RoundResult já é
        snapshot completo. Mas a View passa a conhecer o shape interno
        de CombatantRoundResult.
    Risco: para rítmo, a HUD vai querer "modo do round" (idle, hit,
    miss, combo). Se RoundResult não trouxer isso pré-derivado, a View
    vai precisar de switch + lógica.
    Recomendação:
        a) Computar derivado barato: BattlePhase.decisionForDisplay(player:)
           -> { action, timing, ... } que a View consome.
        b) OU adicionar um RoundDisplaySnapshot derivado em RoundResult,
           sem inflar o modelo de domínio com campos de UI.

M-5. View duplica um resumo "fantasma" para finished
    Arquivo: snap-battle/Features/Battle/BattleView.swift:88-93
    Comportamento atual:
        case .finished: RoundSummaryView(...).hidden()
    Comportamento atual é desnecessário: quando finished, o overlay
    é mostrado; o resumo interno ficar visível ou oculto é detalhe.
    Risco: pequeno; é só uma View duplicada.
    Recomendação:
        Remover case .finished do switch interno; deixar que o overlay
        cuide.

M-6. BattleDemoOpponent.creature é um value estático global
    Arquivo: snap-battle/Features/Battle/BattleDemoOpponent.swift:4-13
    Comportamento atual:
        static let creature = Creature(name: ..., stats: ..., Data())
    É usado como placeholder quando ninguém informa opponent. Pequeno

    Risco: baixo.
    Recomendação:
        a) Mover para Fixtures/, junto com BattleDebugProfile.creature.
        b) Ou exigir opponent não-nulo (mais seguro).

5. ARQUITETURA RECOMENDADA
------------------------------------------------------------------------------

Visão. Manter o motor síncrono e puro. Adicionar (a) uma FSM de domínio
que separe intenção, ritmo, decisão e resolução, e (b) uma camada
sound/AudioProfile que nunca toque o domínio. Tudo por eventos.

5.7 Boundary entre domínio e áudio
----------------------------------------------------------------------------

O domínio sabe APENAS que existe "identidade musical":

struct AudioProfile: Sendable, Equatable {
    let rootNote: MusicalNote           // C, C#, D, ... ou escala numerica
    let scale: Scale                    // major, minor, pentatonicMinor, ...
    let noteRange: ClosedRange<Int>     // MIDI 0..127
    let timbre: Timbre                  // sine, square, saw, sampled(bankID)
    let defaultVelocity: Int            // 1..127
    let defaultDuration: Duration       // beat length
    let motif: [RhythmNote]             // sequência rítmica identidade
}

O domínio gera:
struct MusicalEvent: Sendable, Equatable {
    let note: MusicalNote
    let velocity: Int
    let duration: Duration
    let timbre: Timbre
    let phraseTag: String?              // ex.: "onHit", "onPerfect"
}

A camada de áudio (Services/Audio) consome MusicalEvent e mapeia para
AVAudioEngine/MIDI/sampler. O domínio não importa AVFoundation.

A criatura ganha AudioProfile por construção:
- Material influences timbre (metallic -> saw, stone -> square, etc).
- Labels influence scale.
- Role influences motif.
- Stats (agility) influence default velocity/duration.

(O cálculo é determinístico; pode conviver com DeterministicStatCalculator
estendendo-o para AudioProfile, ou ficar em um novo serviço.)
